# Developer Advisor — 커리큘럼 (v0.1)

> 학습자 프로필: 3년차, 비 IT기업 도메인 기반 Java.
> **약점 축**: ① 인터페이스 개념(왜/언제/어떻게 계약을 정의하나) ② 코드 분리(응집·결합, 책임 나누기).
> **원하는 것**: 레거시 리팩토링 + 기능 추가 훈련, 다양한 도메인으로 상식 확장, 조리 있게 설명하는 능력.
> 모토: **"세상을 거대한 디지털 구조로 본다."**

## 미션의 3요소 (모든 미션 공통)

1. **도메인 브리핑** — 코드 전에 세상 먼저. 이 도메인이 현실에서 어떻게 돌아가는지(와인 등급 체계, 교통 정산망, 결제 청산 구조, 역사적 사건 등) 읽을거리로 제공. → 상식 축적.
2. **코드 미션** — 대부분 **레거시가 주어진다**. 그린필드가 아니라, 냄새 나는 코드를 받아서 ①리팩토링 ②기능 추가를 수행. (java-refactoring-study 방식)
3. **설명 과제** — 설계/도메인을 특정 청자(비개발자 동료, 경영진, 신입)에게 설명하는 글을 작성 → 에이전트가 논리 구조·용어 선택·비유 적절성을 피드백. → 조리 있는 설명 훈련.

## 스테이지 (약점 축을 따라 상승)

11개 스테이지는 3부로 묶인다:
- **제1부 「분리와 계약」** (S1~S4) — 코드 안의 전쟁: 책임을 나누고, 계약을 긋고, 레거시를 길들인다
- **제2부 「구조와 세계」** (S5~S6) — 시스템과 사람: 경계는 코드가 아니라 세계에 긋는 것이다
- **제3부 「현실의 압력」** (S7~S11) — 테스트, 실패, 동시성, 데이터, 단순함: 운영이 가르치는 것들

| 스테이지 | 훈련 주제 | 핵심 질문 |
|---|---|---|
| **S1. 분리의 감각** | 메서드/클래스 추출, 한 덩어리 코드에서 책임 찾기 | "이 코드가 변경되는 이유는 몇 개인가?" |
| **S2. 인터페이스는 계약** | 구현이 아닌 계약에 의존하기, 인터페이스 추출 시점 | "호출자는 무엇을 알아야 하고, 무엇을 몰라야 하는가?" |
| **S3. 의존성 역전** | 포트/어댑터, 외부 세계(DB·API·시간)를 경계 밖으로 | "도메인 로직이 DB 없이 테스트되는가?" |
| **S4. 레거시 길들이기** | 특성화 테스트, 심(seam) 찾기, 안전한 변경 | "무엇이 부서질지 모른 채 어떻게 고치는가?" |
| **S5. 거대한 구조** | 바운디드 컨텍스트, 모듈 경계, 팀 규모의 설계 | "어디서 모델을 갈라야 하는가?" |
| **S6. 구조로 세상 읽기** | 설계 철학, 아키텍처 트레이드오프 에세이 | "이 시스템의 세계관은 무엇인가?" |
| **S7. 테스트가 설계를 이끈다** | 테스트 먼저 쓰기, 테스트하기 어려움 = 설계 냄새 | "테스트가 괴로운 건 테스트 탓인가, 설계 탓인가?" |
| **S8. 실패를 설계하다** | 예외·폴백·멱등성·타임아웃 — 죽는 방법의 설계 | "이 시스템은 어떻게 죽고, 어떻게 다시 일어나는가?" |
| **S9. 동시성의 감각** | async 조합, 경쟁 상태 읽기, 공유 상태의 공포 | "이 코드에 두 명이 동시에 들어오면 무슨 일이 벌어지는가?" |
| **S10. 데이터가 흐르는 길** | 상태와 이벤트, 정합성, 캐시와 사본 | "진실은 어디에 있고, 사본은 언제 거짓말하는가?" |
| **S11. 단순함의 철학** | 깊은 모듈, 복잡도의 출처 추적 | "이 복잡함은 문제의 것인가, 우리가 만든 것인가?" |

**스테이지의 등뼈가 되는 책들** (S7~S11은 각각 대표 책 한 권에 기댄다 — 브리핑이 그 책의 핵심 통찰을 도메인 이야기로 번역):
- S7 ← 켄트 벡 《테스트 주도 개발》 · S8 ← 마이클 나이가드 《Release It!》 · S9 ← 《자바 병렬 프로그래밍(JCiP)》 · S10 ← 마틴 클레프만 《데이터 중심 애플리케이션 설계》 · S11 ← 존 아우스터하우트 《A Philosophy of Software Design》
- 기존 축의 책: S1~S2 ← 마틴 파울러 《리팩터링》 · S3 ← 클린 아키텍처 계열 · S4 ← 마이클 페더스 《레거시 코드 활용 전략》 · S5 ← 에릭 에반스 《도메인 주도 설계》
- S9는 백로그의 async 미션이 첫 재료 (실행 없는 정적 리뷰로 가능한 범위부터 — 블로킹 호출, 타임아웃 부재, 공유 상태). 진짜 멀티스레드 검증은 실행 샌드박스 이후.

- 스테이지는 순차이되 도메인은 교차 가능 (예: S2를 어떤 도메인으로든 재수행).
- 한 스테이지 = 미션 2~4개. 리뷰 점수가 아니라 **"설명 과제까지 통과"** 가 승급 기준.
- 에이전트는 직전 리뷰의 약점을 다음 미션 요구사항에 몰래 다시 심는다 (나선형 반복).

## 도메인 풀 — 스테이지와 독립, 계속 확장

도메인은 스테이지에 묶이지 않는다. 에이전트는 매 미션마다 풀에서 (가급적 안 가본) 도메인을 고르고, 풀 자체도 계속 넓힌다. **어떤 도메인이든 어떤 스테이지의 재료가 될 수 있다.**

🥖 제과·제빵 (베이커스 퍼센트, 발효) · 🎼 음악·합창단 (SATB 파트 배정, 조옮김, 평균율 — 인트로: 카페에서 코딩하던 개발자가 옆 테이블 합창단의 일정 다툼을 듣고 이어폰을 뺀다, "이건 스케줄링 문제잖아") · 🎻 악기 (튜닝, 제작 공정) · 🏗 건축 (건폐율·용적률, 구조 하중) · 🍷 와인 · 🪑 가구 제조 · 🚌 대중교통 · 🚄 철도 (사용자 최애 — 시베리아 횡단 모스크바 시간, 신칸센, 직결 운행) · 💱 금융 (과세, 이자, 청산) · 🛒 커머스 · ✈️ 여행 (항공 운임, 시차) · ☕ 커피 (추출 변수) · 🗺 지도·측량 (좌표계, 투영법) · 📚 도서관 (분류법) · ⚾ 스포츠 기록 (야구 세이버메트릭스) · 🌌 천문 (역법, 시간 체계) · 🏛 역사적 해프닝 (Y2K, 덴버 공항 수하물, 도로명주소 전환) · 🎬 영화 오마주 (설국열차 권한 계층, 마션 자원 계산, 히든 피겨스 마이그레이션) · 🧠 철학 (분류와 경계의 문제) · … 전체 카탈로그: [DOMAIN-POOL.md](DOMAIN-POOL.md). 사용자가 "이 도메인 궁금하다" 하면 즉시 추가.

선택 기준: ① 그 도메인을 **공부해야만 짤 수 있는 규칙**이 있는가(과세, 베이커스 퍼센트, 건폐율처럼) ② 브리핑이 상식으로서 재미있는가 ③ 스테이지의 훈련 주제를 자연스럽게 강제하는가.

## 미션 유형

| 유형 | 흐름 | 주 훈련 |
|---|---|---|
| **리팩토링** | 냄새나는 레거시 제공 → 구조 개선 (동작 보존) | 분리, 인터페이스 추출 |
| **기능 추가** | 레거시 + 신규 요구사항 → 기존 구조를 존중하며 확장 | 개방-폐쇄, 계약 설계 |
| **도메인 로직 구현** | 도메인을 공부해야만 짤 수 있는 로직 (과세, 이자 계산, 환승 정산 규칙 등). 브리핑이 곧 스펙의 절반 — 검증용 정답 수치 제공 | 도메인 규칙의 정확한 코드화, 규칙과 구조의 분리 |
| **설명** | 코드/도메인 → 청자 맞춤 설명문 작성 | 조리 있는 언어화 |
| **설계 리뷰** | 에이전트가 만든 (일부러 결함 있는) 설계를 비평 | 비판적 읽기 |

## 난이도와 범위 — 두 개의 독립 축

**난이도 (Easy / Normal / Hard)** — 도메인 규칙의 복잡도, 요구사항의 모호함 양, 함정의 개수. "머리가 얼마나 아픈가."

**범위 (단일 파일 / 여러 파일 / 모듈 경계)** — 변경이 미치는 구조적 스코프. "손이 얼마나 넓게 가는가."

- **단일 파일** — 메서드·클래스 안에서의 책임 정리.
- **여러 파일** — 파일(클래스) 사이의 경계와 계약. 계층 구조(3-tier) 포함.
- **모듈 경계** — 모듈/애플리케이션 사이의 책임 분리. 멀티모듈, 서비스 분리, 바운디드 컨텍스트.

두 축은 직교한다: **Hard + 단일 파일**(대중교통 FareCalc — 한 파일이지만 지옥), **Easy + 모듈 경계**(규칙은 단순하나 어디에 둘지가 문제) 모두 존재한다. 학습자는 두 축을 독립적으로 올린다. 엔진 코드 원칙은 범위와 무관하게 유지된다.

## 질문 창과 평판 — 소통도 실력이다

미션 화면의 채팅 창에서 에이전트(선배 개발자/기획자 역할)와 소통한다. 요구사항의 **의도적 모호함은 여기서 되물어야 풀린다.**

- 대화 기록은 리뷰 시 **평판**으로 평가된다 — 루브릭 점수(채점)와는 별도의 정성 평가. 예: "질문 수준: 3년차 중상. 예산 규칙의 모호함을 구현 전에 짚었고, 재현 예시를 곁들였음."
- 평가 기준: ① 타이밍 — 모호함을 구현 전에 짚었나 ② 구체성 — 예시·경계값을 들었나 ③ 자기 시도 — 물어보기 전에 뭘 해봤나 ④ 매너.
- **질문이 많다고 감점되지 않는다.** 좋은 질문은 가점, 물어봐야 할 것을 안 물은 채 임의 확정한 것이 감점.
- 모델 운영: 채팅 응답은 Haiku(빠르고 저렴, 실시간 대화), 리뷰·평판 평가는 상위 모델이 대화 전체를 읽고 수행. `LlmClient` 추상화에 역할별 모델 설정을 둔다.

## 히든 케이스 — 현실은 명세보다 더럽다

모든 미션에는 요구사항에 없는 **히든 케이스**가 2~3개 숨어 있다 (미션 화면에는 개수만 표시, 리뷰에서 공개). 현실의 입력은 명세를 읽지 않기 때문이다.

- 유형: 정수 오버플로(21억을 넘는 설비 ID), 예상 밖 타입/오타("G0LD"), 0과 음수(0개월 계약, 대지면적 0), 경계의 시간(자정을 넘는 환승).
- 평가: 히든 케이스를 못 막은 것 자체보다, **막을 수 없는 입력이 온다는 전제가 코드에 있는가**(검증 계층, 명시적 실패)를 본다.
- 각 히든 케이스에는 가능하면 실제 사고 사례를 붙인다 (Ariane 5 오버플로, 2038 문제 등) — 고생담이 곧 교보재.

## 유머 원칙 — 한 스푼, 도메인에서

콘텐츠에는 유머를 넣는다. 단 규칙이 있다:

1. **정해진 자리는 없다** — "제약 마지막엔 반드시 농담 하나" 같은 슬롯 규칙 금지 (실제로 그렇게 운영하다 사용자에게 지적받았다). 유머는 자연스럽게 떠오른 자리에만 넣고, 억지스러우면 뺀다. 유머가 하나도 없는 미션도 정상이다.
2. **정보를 해치지 않는다** — 검증 수치와 규칙 본문은 건드리지 않는다.
3. **도메인에서 나온 유머만** — 억지 말장난 대신 그 세계의 진실에서 나오는 건조한 유머. (예: "시험 중 시음은 허용됩니다. 단, 커밋은 취하기 전에.")
4. **고생담은 최고의 유머다** — "1997년 김부장의 주석"처럼, 레거시 코드의 비극은 그 자체로 웃기다.
5. 시간 제한 같은 메타 장치도 유머가 된다 — KTX 미션의 제한시간은 서울→부산 소요시간(약 160분).

## 감성 원칙 — 정서는 장식이 아니라 문제의 일부다

유머와 짝을 이루는 규약. 콘텐츠에는 정서적 순간을 넣되, 반드시 도메인 안에서 나와야 하고, 가능하면 **풀어야 할 문제 자체에 스며들어야 한다**.

- 기준 사례: *"통일호는 KTX 개통 하루 전인 2004년 3월 31일에 퇴역했습니다. 코드에는 아직 살아 있지만요."* — 이 문장은 분위기이면서 동시에 미션의 질문이 된다: **그 죽은 분기를 지워도 되는가?** 기술적 정답(데드 코드 제거)과 시스템의 지층으로서의 코드 사이의 갈등. 꼬리 질문·설명 과제의 재료.
- 다른 예: 시나리오의 "말없이 떠나는 사용자는 분기 지표로만 돌아온다"(침묵 버그의 공포), 히스토리 빈 화면의 "세상은 여전히 if문 범벅인 채로 당신을 기다리고 있습니다".
- 금지: 주제와 무관한 신파, 교훈 강요, 감성을 위한 감성. 정서는 항상 사실(퇴역 날짜, 실제 사건)에 뿌리내린다.
- 활용 위치: 브리핑의 마지막 단락, 시나리오 에필로그, 꼬리 질문, 결말 티저.

### 히든 퀘스트 — 어떤 문장은 힌트였습니다

감성 원칙의 게임화. **요구사항 어디에도 없는 과제를 지문에 문장으로만 심는다.** 힌트로 알려주지 않는다.

- 구조: `hiddenQuest { plant(심은 곳), condition(달성 조건), revealOnSuccess, revealOnMiss }` — 미션 화면에는 아무것도 표시되지 않는다. 지문의 문장이 유일한 단서.
- 리뷰에서만 공개: 달성 시 "알아채셨나요 — 지문의 마지막 문장을." / 미달 시 "요구사항에 없었으니 감점은 아닙니다. 다만 어떤 문장은 힌트였습니다. 다음 지문은 조금 다르게 읽히실 겁니다."
- 기준 사례: KTX 미션 — 브리핑 마지막 문장("통일호는 개통 하루 전 퇴역했습니다. 코드에는 아직 살아 있지만요")과 레거시의 통일호 분기. 시키지 않아도 그 분기를 정리하고 흔적을 남기면 달성.
- 원칙: 감점 없음(보너스 서사만), 히든 케이스(입력 방어)와 구분 — 퀘스트는 **읽기의 보상**이다.

## 결말 분기 — 이 코드의 가능한 미래들

미션을 받을 때 게임 상태창처럼 **결말 분기**가 함께 제시된다. 리뷰의 시나리오는 그중 어느 결말에 도달했는지 도장을 찍고 그 이야기를 서술한다.

| 결말 | 의미 | 티저 문체 |
|---|---|---|
| ☕ **평온** | 최상 — 아무 일도 일어나지 않는다. 운영에서 그것은 최고의 찬사다 | "당신의 코드를 아무도 이야기하지 않는다" |
| 🔧 **핫픽스** | 보통 — 돌아는 간다. 가끔 새벽에 몰래 고친다 | "커밋 메시지에 'fix'가 늘어간다" |
| 🚨 **새벽 3시** | 최악 — 장애 회고 문서에 커밋 해시가 박제된다 | "전화벨은 항상 금요일 밤에 울린다" |
| 🔒 **???** | 히든 — 달성 조건 비공개 (예: 히든 케이스 전부 방어 + 모호함 전부 되묻기) | 잠김 |

- 미션 화면: 4개 결말이 상태창으로 표시, 히든은 🔒 잠김.
- 리뷰 화면: 시나리오 섹션에 "달성한 결말" 배지. 히든 달성 시 잠금 해제 연출.
- 티저는 도달 조건의 힌트이되 답안지가 아니다 — "무엇을 하면"이 아니라 "무슨 일이 벌어지면"으로 쓴다.

## 엔진 코드 원칙 — "바깥은 준다"

미션에는 두 종류의 코드가 있다:

- **작업 대상** (레거시/뼈대) — 학습자가 고치거나 채운다.
- **제공 코드 (엔진)** — 인프라 내부 구현은 항상 완성된 채로 제공된다. 예: `MapDb.insert()`는 딕셔너리에 저장하는 수준으로 이미 구현되어 있다.

이유: 학습자의 연습 대상은 **인터페이스 정의(엔진을 어떤 계약 뒤에 둘 것인가)**이지, `insert()` 내부 구현이 아니다. 현업에서도 DB 드라이버를 직접 짜지 않는다. 루브릭은 "도메인 로직이 엔진을 직접 아는가"를 평가한다.

### 외부 경계는 코드로 제공한다

미션의 시스템이 외부(다른 팀·다른 시스템)와 맞닿는 지점은 말로 설명하지 않고 **코드로 제공**한다. 단, 그 경계의 품질은 미션의 시대상을 따른다:

- 이상적: Java `interface` (신규 시스템, 잘 관리된 경계)
- 레거시 현실: 규약이 엉터리인 `public` 메서드 덩어리, **DB 폴링**(테이블 스키마가 곧 계약 — 5분마다 SELECT 하는 타 시스템), 파일 드롭, 전역 `static` 변수 공유
- 경계의 품질 자체가 문제의 일부다: "이 public 메서드 뭉치를 어떤 인터페이스로 승격시킬 것인가", "저 테이블을 폴링하는 시스템이 있는 한 컬럼을 함부로 못 바꾼다"는 것을 발견하는 것이 훈련.

## 기획자 모드 — 같은 문제, 다른 의자

브리핑 이후 미션은 **모드**를 선택할 수 있다 (미션별 활성화 — 어울리지 않는 도메인은 개발자 모드 전용).

| 모드 | 정체 | 진행 | 평가 |
|---|---|---|---|
| 💻 **개발자 모드** | 기존 흐름 | 코드 구현 → 제출 | 루브릭 + 히든 케이스 + 결말 |
| 🤝 **기획자 · 회의 모드** | IT 시스템 기획자로서 실시간 회의. 에이전트가 이해관계자들(개발 리드/재무/법무/운영)을 연기 — 각자 **공개 입장 + 비공개 관심사** 보유 | 채팅으로 협상·챌린지 → **합의문 제출** | 비공개 관심사를 캐냈는가, 합의의 실행가능성, 설득 방식. 모델: Sonnet급 |
| 📋 **기획자 · 검토 모드** | 다차원 검토서 작성. 미션이 관점 축(재정/시스템/법률/운영/리스크)을 제시 | 트레이드오프를 명시한 **검토서 작성 → 제출** | 전용 루브릭: 이해관계자 커버리지, 트레이드오프 명시, 실행가능성, 설득력 |

원칙:
- 기획자는 **시스템 기획자**다 — 기존 시스템 개선, "A팀/A시스템이 하던 일을 B로 옮기기"(경계 재설계) 같은 챌린지가 주 무대. 코드 없이 하는 바운디드 컨텍스트 훈련.
- **"개발하지 않는 것"이 정답일 수 있다** — 프로세스 변경/정책으로 풀리는 문제를 개발로 풀면 감점되는 미션 설계 가능.
- 비공개 관심사는 히든 케이스의 기획 버전 — 회의에서 캐내지 못하면 합의가 나중에 부러진다 (시나리오 에필로그에 반영).
- 미시적 기획도 환영 (버튼 하나의 정책, 정산 마감 시각 변경 등).
- 플래그십: 🚄 KTX 미션 (재정·시스템·법률이 실제로 충돌한 사건).

## 프로젝트 모드 — 맨땅에서 (from scratch)

미션이 "물려받은 코드"라면, 프로젝트는 **아무것도 없는 곳**에서 시작한다. 별도 UX(여정 지도)로 접근하며, 하나의 큰 목표가 **소미션 체인**으로 갈라진다.

- **여정 지도 UX**: 소미션 노드가 세로로 이어진 캠페인 맵. 제출해야 다음 노드 해금. 완료 ✓ / 진행 중 ▶ / 잠김 🔒.
- **소미션 체인**: 도메인 모델 → 핵심 로직 → 정책 → 저장 경계 → 확장 → 설명 보고서 순의 점증 구조. 각 소미션은 이전 제출물 위에 쌓인다.
- **핵심 반전 — 어제의 코드가 오늘의 레거시다**: 소미션 N의 리뷰는 소미션 1~N-1에서 학습자가 직접 쓴 코드를 근거로 삼는다. "이 결합은 소미션 2에서 당신이 만든 것입니다"가 최고의 교육 순간. 커리큘럼 전체 명제(레거시는 남이 만든 게 아니라 과거의 내가 만든 것)의 체험판.
- 미션의 장치들(히든 케이스, 결말 분기, 질문 창, 평판)은 프로젝트에도 적용 가능하되, v1은 가볍게 시작.
- 플래그십: 🚲 공공자전거 시스템 (대여/반납, 요금, 재배치 — 자전거 쏠림은 실제 운영의 최대 난제).

- **지금 (프로토타입~M3)**: Claude(이 세션의 에이전트)가 미션·브리핑·리뷰를 생성해 저장.

## 이 프로젝트 자체가 S2~S3 실습이다

Developer Advisor 백엔드를 만들 때 학습자가 직접 참여하는 실습 지점:

- `LlmClient` 인터페이스 설계 → **S2 인터페이스는 계약** 실전
- Claude/Ollama 어댑터 분리 → **S3 의존성 역전** 실전
- 프론트 `modules/missions` 격리 → **S1 분리의 감각** 실전

→ 진행 방식: Claude가 구조와 지시를 주고, 핵심 인터페이스/클래스는 학습자가 구현 → Claude가 리뷰.
