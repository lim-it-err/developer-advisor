# 실습 미션 — LlmClient 인터페이스 설계 (S2 · S3)

> 이 실습은 developerAdvisor 프로젝트 자체가 커리큘럼이라는 원칙(CURRICULUM.md)에 따라 만들어졌습니다.
> `service/` 백엔드에는 이미 동작하는 레퍼런스 구현이 있지만, **먼저 열어보지 마세요.** 아래 과제를 스스로 끝낸 뒤에
> `service/src/main/java/kr/co/workaround/advisor/application/port/LlmClient.java`와
> `service/src/main/java/kr/co/workaround/advisor/adapter/out/llm/` 를 열어 비교하는 것이 이 미션의 완주입니다.

## 상황

advisor 서비스는 세 개의 LLM 백엔드를 갈아 끼워야 합니다: 로컬 테스트/키 없는 개발용 **mock**, 무료로 로컬에서 돌리는 **Ollama**,
실제 품질이 필요한 **Claude**. 도메인 서비스(`MissionService`, `ReviewService`, `ChatService` 등)는 지금 어떤 provider가
붙어 있는지 알아서는 안 됩니다. 프로바이더를 바꾸는 순간에도, 도메인 서비스 코드는 한 줄도 바뀌지 않아야 합니다.

이 서비스는 세 가지 종류의 결과물을 LLM에게 요청합니다:
1. **미션 생성** — 브리핑, 시나리오, 레거시 코드, 루브릭, 히든 케이스, 결말 분기 등을 담은 꽤 복잡한 구조.
2. **제출물 리뷰** — 점수, 근거 인용, 평판, 다음 결말 도장 등을 담은 또 다른 복잡한 구조.
3. **미션 중 채팅 응답** — 짧은 텍스트 한 줄.

세 요청 모두 "그냥 문자열"이 아니라 **정해진 형태의 데이터**를 원합니다. 그리고 역할마다(생성/리뷰/채팅) 어떤 provider와
모델을 쓸지 설정으로 결정하고 싶습니다(예: 채팅은 저렴하고 빠른 모델, 리뷰는 품질 좋은 모델).

## 과제 1 — 인터페이스를 먼저 설계하라 (S2: 인터페이스는 계약이다)

레퍼런스 코드를 열기 전에, 아래 질문에 스스로 답하며 자기만의 `LlmClient` 인터페이스 초안을 작성하세요.

- **시그니처**: 메서드는 몇 개여야 할까요? "미션 생성해줘"와 "채팅 답장해줘"를 서로 다른 메서드로 나눌까요, 아니면 하나의
  메서드로 통일할까요? 나눈다면 무엇을 기준으로 나눌지, 통일한다면 "어떤 종류의 요청인지"를 어떻게 표현할지 정하세요.
- **구조화된 출력을 어떻게 표현할 것인가**: 서비스 코드가 "미션 생성 결과"를 받을 때, 그 결과가 `String`(JSON 텍스트)이면
  서비스가 직접 파싱해야 합니다. 그게 싫다면 무엇을 인터페이스에 넣어야 호출자가 "원하는 타입"을 미리 선언할 수 있을까요?
  (힌트: 자바에서 "이 타입으로 돌려줘"를 표현하는 방법이 메서드 인자로 넘길 수 있는 게 몇 가지 있습니다.)
- **역할별 모델을 어디서 결정할 것인가**: "생성은 Claude Sonnet, 채팅은 Haiku" 같은 매핑은 어느 레이어의 책임일까요?
  인터페이스를 호출하는 서비스 코드가 provider 이름이나 모델 이름 문자열을 알아야 할까요, 아니면 "이건 리뷰 작업이야"
  정도만 알려주고 나머지는 인터페이스 뒤에서 알아서 처리해야 할까요?

**산출물**: 실제 컴파일 가능한 `.java` 인터페이스 초안 (구현은 아직 없어도 됩니다). 짧은 메모로 위 세 질문에 대한
당신의 결정과 그 이유를 남기세요.

## 과제 2 — 어댑터를 갈아 끼우며 도메인이 흔들리는지 확인하라 (S3: 의존성 역전)

과제 1에서 만든 인터페이스를 기준으로, 순서대로 어댑터를 추가하세요:

1. **mock 어댑터** — 고정된 값을 반환하는 가장 단순한 구현. 여기서부터 시작해야 도메인 서비스를 키 없이 돌려볼 수 있습니다.
2. **Ollama 어댑터** — 로컬 서버(`http://localhost:11434`)에 HTTP로 요청을 보내는 구현. Ollama는 JSON 모드를 요청해도
   완벽한 JSON을 보장하지 않는다는 점을 어떻게 다룰지 고민해 보세요.
3. **Claude 어댑터** — Anthropic Messages API에 요청을 보내는 구현. "자유 텍스트로 응답받고 파싱"과 "도구(tool) 호출을
   강제해서 구조화된 입력을 받기" 중 어느 쪽이 신뢰도가 높을지 비교해 보세요.

각 어댑터를 추가할 때마다 **self-check**: `MissionService`, `ReviewService`, `ChatService` 등 도메인/애플리케이션
코드를 열어서, 정말 한 줄도 바뀌지 않았는지 확인하세요. 만약 바뀐다면— 어디가, 왜 바뀌었나요? 그 지점이 바로 "인터페이스가
새지 않았다면 나오지 않았을 결합"입니다.

- 바뀐 부분이 있다면: 그 코드는 provider의 무엇을 알아버린 걸까요 (모델 이름? 응답 포맷? 에러 종류?)
- 바뀐 부분이 없다면: 왜 없었는지 — 인터페이스의 어떤 설계 선택이 그걸 가능하게 했는지 설명해 보세요.

## 비교 단계 — 레퍼런스와 대조하기

과제 1, 2를 마친 뒤에 레퍼런스를 여세요. 레퍼런스는 다음과 같은 형태입니다(시그니처만 공개):

```java
public interface LlmClient {
    <T> T complete(LlmRole role, String prompt, Class<T> type);
}
```

그리고 `adapter/out/llm/`에 `mock/`, `claude/`, `ollama/` 세 개의 어댑터 패키지가 있고, 그 뒤에 `RoutingLlmClient`라는
내부 라우터가 `LlmRole → {provider, model}` 설정을 보고 알맞은 어댑터에 위임합니다.

당신의 설계와 비교하며 서술하세요 (코드가 아니라 **말로 설명**하는 것이 통과 기준입니다):

- 레퍼런스는 프롬프트를 `String`으로 받습니다. 당신은 전용 타입(`PromptTemplate` 같은)을 만들었나요, 아니면 `String`을
  그대로 썼나요? 각각의 장단점은 무엇인가요? (힌트: 타입 안전성 vs. 단순함, 슬롯 치환 로직을 어디에 둘 것인가)
- 레퍼런스는 `Class<T> type`으로 구조화된 출력을 표현합니다. 이 방식이 "제네릭 메서드 + Class 토큰"이라는 자바의 흔한
  패턴인데, 왜 `Object`를 받아서 캐스팅하게 하지 않았을까요? 컴파일 타임에 무엇을 얻고 무엇을 포기했나요?
- 레퍼런스는 `LlmRole` enum(GENERATE/REVIEW/CHAT) 하나로 "어떤 요청인지"와 "어떤 provider/model을 쓸지"를 분리했습니다.
  당신의 설계에서 이 두 가지는 어떻게 분리되어 있나요(혹은 분리되어 있지 않나요)? 역할이 4번째, 5번째로 늘어난다면
  당신의 설계와 레퍼런스 설계 중 어느 쪽이 더 적은 변경으로 버틸까요?

## 통과 기준

이 실습은 코드 리뷰가 아니라 **설명 과제**로 통과합니다. 다음 질문에 스스로(혹은 리뷰어에게) 명료하게 답할 수 있으면
완료입니다:

- "왜 이 계약(시그니처)인가?" — 대안을 최소 하나 이상 검토했고, 선택하지 않은 이유를 설명할 수 있는가.
- "도메인 로직이 provider를 직접 아는가?" — mock → Ollama → Claude로 어댑터를 바꾸는 동안 도메인 계층 코드가
  실제로 몇 줄 바뀌었는지 정확히 말할 수 있는가.
